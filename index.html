Summary — Two Strong Paths

Path A
Traditional WordPress
PHP · Page builders · Plugins · WP Rocket · Cloudflare
Strengths
  • Both devs fully productive from day one — zero ramp-up cost during the rebuild
  • WP specialist contributes fully across page building, content, plugin config, and theme work
  • Seamless agency handoff — team already speaks the language of the codebase
  • Done properly (lean theme, WP Rocket, Cloudflare, strict plugin discipline), realistic Lighthouse target of 85–92 — clears all real-world Core Web Vitals thresholds
  • Lowest build risk — familiar tools mean fewer surprises and faster delivery
  • Full plugin ecosystem for future features — forms, search, integrations all available out of the box
Tradeoffs
  • Performance ceiling exists — bloat is managed, not eliminated. One new plugin can regress scores.
  • Multilingual via WPML adds $99–199/yr licensing, plugin conflict risk, and 6× content duplication workload for the team
  • Security patching is ongoing — WP core, plugins, and PHP all require regular attention
  • Performance drift is likely as the site grows — same conversation may resurface in 18–24 months
Path B
Headless CMS + Next.js
Sanity · Next.js / React · Vercel · Formspree · next-seo
Strengths
  • Performance solved architecturally — 95–100 Lighthouse, sub-second LCP, doesn't erode over time as the site grows
  • Multilingual built in natively — one document per page holds all 6 languages, no WPML, no duplication, no licensing fee. Sanity AI Assist auto-translates for editors to review.
  • Full-stack dev owns and maintains the Next.js codebase independently post-handoff
  • WP specialist manages all content via Sanity Studio from day one — no dev involvement needed for content updates
  • No PHP attack surface on the public site — security maintenance drops significantly
  • Future-proof — API-first content reusable across web, mobile app, email, and any future channel
Tradeoffs
  • Sanity Studio is a structured editor, not WYSIWYG — WP specialist loses the familiar page-building experience (Storyblok is an alternative with visual editing if this is a blocker)
  • WP specialist's dev contribution is initially limited to content — growing into component work takes time
  • Full-stack dev becomes sole owner of the codebase — no dev redundancy if unavailable
  • i18n schema must be designed from day one — needs to be explicitly in scope in the agency brief
The core question to answer: Both paths are legitimate for this team. The decision comes down to what matters most — team continuity and equal contribution from both devs (favours Traditional WP), or permanent performance ceiling and cleaner multilingual support (favours Headless CMS). Headless WP (Path B in the comparison above) carries the highest complexity with the weakest handoff outcome for this team — it combines the costs of both systems without the full benefits of either.
Regardless of which path is chosen — things to lock into the agency brief:
  • 6-language multilingual support must be in scope from day one, not retrofitted later
  • Full documentation and a codebase walkthrough session for the internal team before final payment
  • If headless: specify Sanity + Next.js explicitly — not just "headless CMS" — and confirm i18n schema is designed upfront
  • If WordPress: specify performance targets (85+ Lighthouse on key pages) as an acceptance criterion, not an aspiration
